# Forge Knowledge Base

The Knowledge Base is the central repository for organizational knowledge that feeds into the Forge Learning Loop. It stores system profiles, auto-generated documentation, and convention mining reports that skills and agents consume to deliver context-aware assistance.

## Directory Structure

```
knowledge-base/
  profiles/         # System and service profiles
  generated-docs/   # AI-generated documentation (from doc-generation workflow)
  conventions/      # Convention mining reports (from convention-sync workflow)
```

## Profiles (`profiles/`)

System profiles describe individual services, libraries, or subsystems in a structured format. They are consumed by skills to provide domain-specific context.

**Profile format:**

```yaml
# profiles/order-service.yaml
name: order-service
type: microservice
language: kotlin
framework: spring-boot-3
database: postgresql
dependencies:
  - inventory-service
  - payment-service
  - notification-service
api_style: rest
auth: jwt
description: |
  The Order Service handles order lifecycle management including
  creation, payment processing, fulfillment tracking, and cancellation.
key_patterns:
  - saga-pattern for distributed transactions
  - event-sourcing for order state changes
  - circuit-breaker for downstream calls
```

**When to add a profile:**
- When a new service or major component is created
- When migrating a legacy system that agents will interact with
- When a service's architecture changes significantly

## Generated Documentation (`generated-docs/`)

This directory contains documentation automatically generated by the `doc-generation` CI workflow. Files here are produced by analyzing code, commit history, and existing documentation to fill coverage gaps.

**Conventions for generated docs:**
- Files are named after the source they document (e.g., `order-service-api.md`)
- Each file includes a `Generated:` timestamp in its header
- Generated docs are regenerated on PR merges that affect the documented code
- Manual edits to generated docs will be overwritten; use `profiles/` instead for curated knowledge

## Conventions (`conventions/`)

Convention mining reports capture coding patterns, architectural decisions, and team practices discovered by analyzing the codebase. The `convention-sync` workflow runs monthly to detect new patterns and flag drift between documented conventions and actual code.

**Convention report format:**

```yaml
# conventions/naming-conventions.yaml
category: naming
last_mined: 2026-01-15
confidence: 0.92
patterns:
  - pattern: "Controllers end with 'Controller'"
    occurrences: 47
    exceptions: 2
    recommendation: enforce
  - pattern: "Service interfaces use 'Service' suffix, implementations use 'ServiceImpl'"
    occurrences: 23
    exceptions: 0
    recommendation: enforce
  - pattern: "DTOs use 'Request'/'Response' suffix"
    occurrences: 31
    exceptions: 5
    recommendation: suggest
```

## Contributing

### Adding a New Profile

1. Create a YAML file in `profiles/` following the format above
2. Include all required fields: `name`, `type`, `language`, `description`
3. List key patterns and dependencies accurately
4. Submit a PR for team review

### Reviewing Generated Docs

Generated docs are automatically created but benefit from human review:
1. Check accuracy of API descriptions
2. Verify dependency relationships
3. Flag outdated information by adding comments to the source profiles

### Interpreting Convention Reports

Convention reports use confidence scores:
- **0.9+**: Strong pattern, safe to enforce
- **0.7-0.9**: Common pattern, recommend following
- **0.5-0.7**: Emerging pattern, worth discussing
- **Below 0.5**: Weak signal, needs more data

## Integration with Skills

Skills reference knowledge base content through the `forge-knowledge` MCP server:

```markdown
## Instructions
When working on the order service, first retrieve its profile:
- Use `forge-knowledge.search-profiles` with query "order-service"
- Review the key_patterns listed in the profile
- Follow the conventions from the latest mining report
```

## Maintenance Schedule

| Task | Frequency | Triggered By |
|------|-----------|-------------|
| Convention mining | Monthly | `convention-sync` workflow |
| Doc generation | On PR merge | `doc-generation` workflow |
| Profile review | Quarterly | Manual review cycle |
| Staleness check | On code change | `domain-skill-staleness` workflow |
