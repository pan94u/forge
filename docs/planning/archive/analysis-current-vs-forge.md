## 对比维度一：速度

**当前过程（Claude Code CLI）的速度是碾压级的。**

| | 当前过程 | 如果用 Forge |
|---|---|---|
| Phase 0 骨架 | ~10 小时出 244 文件 | 不适用——Forge 是给业务交付用的，不是给平台搭建用的 |
| 灵活性 | 随时切换：Agent 失败 → 手动写 → 再起 Agent | 需要走 OODA → 底线 → HITL 流程 |
| 并行度 | 一次启动 6 个 Agent | SuperAgent 是单实例串行 OODA |
| 冷启动 | 直接开干 | 需要先部署 MCP Server、配置 Skill、建立知识库 |

**结论：对于"从零建平台"这类创造性工作，当前过程更快。** Forge 的流程有 overhead——部署、配置、审批——这些在创建型工作中是摩擦。

---

## 对比维度二：质量

**这是当前过程的最大软肋。**

logbook 记录的实际质量问题：

| # | 问题 | 何时发现 | Forge 能否防止 | 怎么防 |
|---|------|---------|---------------|--------|
| 1 | **CORS 通配符** `setAllowedOrigins("*")` | Session 2 手动发现 | **能** | `security-baseline.sh` 自动扫描 |
| 2 | **JDK 版本不一致** 7 个文件用 17 而非 21 | Session 2 Agent 发现 | **能** | `gradle-build` Skill 强制版本 + `code-style-baseline` |
| 3 | **假流式** 50 字符分块 + Thread.sleep(20) | Session 3 规划时发现 | **部分能** | `testing-standards` Skill 会标记为 stub；但 baseline 不一定能检测 |
| 4 | **MCP 端点路径不匹配** `/tools/list` vs `/mcp/tools/list` | Session 3 规划时发现 | **能** | `api-contract-baseline.sh` 接口一致性校验 |
| 5 | **Explore Agent 误判** 报告 3 个 baseline 缺失 | Session 2 手动验证 | **能** | `codebase-profiler` 做精准画像，不依赖单次 grep |

5 个已知质量问题中，Forge 能预防 4-5 个。

**更严重的是：我们不知道还有多少未发现的问题。** 38,600 行代码，实际 review 的可能不到 10%。当前过程没有系统性的质量检查——全靠人"碰巧发现"。

## 对比维度三：知识持久性

| | 当前过程 | Forge |
|---|---|---|
| **跨 Session 记忆** | 上下文压缩 → 细节丢失；logbook 是手动补救 | knowledge-base 自动沉淀 |
| **决策理由** | 散落在对话中，session 结束后难追溯 | ADR 结构化存储 |
| **教训沉淀** | logbook 记了 6 条"关键教训"——但纯靠人工 | 进化环自动提取 + skill-feedback-analyzer |
| **上下文恢复** | 新 session 靠 summary 恢复，信息有损 | MCP 查询实时获取最新状态 |

**实际案例**：Session 3 就是因为上下文压缩而重启的。需要重新读 logbook、重新读源码来恢复状态。如果有 Forge 的知识库 + MCP，直接查询即可。

## 对比维度四：Agent 可靠性

logbook 暴露的真实可靠性：

```
12 个 Agent 执行记录：
  ✅ 完全成功: 8 个 (67%)
  ⚠️ 部分完成: 2 个 (rate limit)
  ❌ 完全失败: 2 个 (权限/token limit)

后果：
  - Agent ac247ab 失败 → 手动写 27 个文件（~2 小时）
  - Agent aebd09e 重试失败 → 放弃该路径
  - Agent a3d0b68 误判 → 差点创建已存在的文件
```

Forge 的 SuperAgent OODA 循环本质上就是为了解决这个问题——底线检查失败 → 自动回到 Observe → 重新尝试。而不是"Agent 失败 → 人工兜底"。

## 对比维度五：适合的场景

| 场景 | 当前过程更优 | Forge 更优 | 原因 |
|------|-----------|-----------|------|
| **从零创建项目骨架** | **是** | | 创造性强，需要极致灵活性，质量要求可以后补 |
| **增量功能开发** | | **是** | 规范一致性、底线检查、知识上下文都有价值 |
| **跨栈迁移** | | **是** | 知识提取 + 规范强制 + 行为验证 |
| **多人协作** | | **是** | 统一规范、共享知识、结构化审查 |
| **长期维护** | | **是** | 进化环、知识积累、convention-sync |
| **快速原型** | **是** | | Forge 的流程 overhead 拖慢原型速度 |
| **一次性脚本/工具** | **是** | | 杀鸡焉用牛刀 |
| **生产级业务代码** | | **是** | 质量底线不可妥协 |

## 总结：真实的优劣

### 当前过程（Claude Code CLI）的真实优势

1. **极致速度**：38K 行 / 10 小时，没有任何流程 overhead
2. **极致灵活**：Agent 失败立刻手动接管，不需要等流程
3. **零部署成本**：开箱即用，不需要 MCP Server / 知识库 / 底线脚本
4. **适合创造**：探索性工作、原型、骨架——"先出来再说"

### 当前过程的真实劣势

1. **质量靠运气**：5/38600 行的已知 bug 率可能只是冰山一角
2. **知识是一次性的**：上下文压缩 = 遗忘；每个 session 都在"重新理解"
3. **Agent 不可靠**：33% 的失败/部分完成率，需要人工兜底
4. **没有安全网**：CORS 通配符这种安全漏洞直接进了 commit
5. **不可复制**：换一个人来继续开发，需要从头理解全部 244 个文件

### Forge 平台的真实优势

1. **质量有底线**：不管谁写的代码，baseline 必须过
2. **知识可积累**：用得越多越好，不是每次从零开始
3. **流程可重复**：OODA + HITL = 结构化的交付流程
4. **适合团队**：规范统一、知识共享、结构化审查

### Forge 平台的真实劣势

1. **启动成本高**：需要先建 Skill、部署 MCP、配置底线——这本身就是一个项目
2. **灵活性降低**：OODA → 底线 → HITL 流程对"快速试错"是摩擦
3. **单线程**：SuperAgent 单实例 OODA，不能像当前一样 6 个 Agent 并行
4. **维护成本**：Skill 要保鲜、MCP Server 要运维、底线要更新

---

## 一句话结论

> **Claude Code CLI 是跑车——极快、灵活、但没有安全气囊。**
> **Forge 是装甲运兵车——有流程、有保障、有记忆，但启动慢、转弯半径大。**
>
> 做原型用跑车，上战场用装甲车。我们现在用跑车造装甲车——这是对的。

## 对规划的反馈

这个分析对 Phase 1 实施计划的启示：

| # | 启示 | 建议 |
|---|------|------|
| 1 | 当前过程的质量问题证明了底线体系的必要性 | Phase 1 WP7（测试）不应降低优先级 |
| 2 | 并行 Agent 的速度优势是 Forge 缺失的 | 考虑 SuperAgent 未来支持并行 OODA（多模块同时迁移） |
| 3 | logbook 是手动的知识沉淀替代品 | 验证了 execution-logger + knowledge-base 的设计方向 |
| 4 | 上下文压缩是真实痛点 | 验证了 MCP 知识集成的价值——让 AI 不依赖对话历史 |
| 5 | "用跑车造装甲车"是正确策略 | Phase 0-1 用 Claude Code CLI 快速搭建，Phase 2+ 开始 dogfooding |
