# Session 5 设计回顾 — 从"设计保护"到"平台能力提炼"

> Plan Mode 讨论全记录（2026-02-18）
>
> 本文档记录了 Phase 1.5 Docker 部署成功后，在 Plan Mode 中进行的一次深度设计讨论。
> 这次讨论从"如何保护已验证的设计"出发，逐步演化为"如何将开发者痛点编码为平台能力"，
> 最终产出了 Baseline v1.3 的核心设计思想和实施计划。

---

## 背景：讨论发生的时刻

Phase 1.5 的 Docker 部署刚刚完成。经历了 90 分钟的艰苦调试（13 次构建尝试，10 个计划外问题），三个容器终于全部 healthy，`http://localhost:9000` 返回 200。用户在浏览器中试用了运行中的应用。

这是一个微妙的时刻——**成功带来了喜悦，但也立刻触发了焦虑**。

---

## 第一个问题：设计会被遗忘

### 用户的担忧

> "我试用了下现在感觉非常好，特别是 UI 交互和我们现在的设计，我很担心未来会再演化过程中对当前的设计细节有所遗忘，即便他是正确的和已经被验证过的，我们如何规避这类事情呢"

用户看到了运行中的应用——Monaco Editor + AI Chat sidebar + File Explorer 的布局，流式响应的交互体验，路由的组织方式——这些都是经过多次迭代验证通过的设计。但软件的本质是不断演化的，而**演化过程中最容易丢失的，恰恰是那些"看起来理所当然"的正确设计**。

### 探明范围：哪些层面需要保护？

**问题**：你担心遗忘的「设计细节」主要是哪些层面？

| 选项 | 说明 |
|------|------|
| UI/UX 交互设计 | 组件布局、交互流程、样式细节等前端设计决策 |
| 架构与 API 契约 | 后端架构、API 接口、数据模型等技术设计 |
| 产品功能逻辑 | 功能行为、业务规则、边界条件等产品决策 |
| 全部都担心 | 以上所有层面都希望有保护机制 |

**用户选择**：全部——UI/UX 交互设计、架构与 API 契约、产品功能逻辑、全部都担心。

### 现状分析：已有 vs 缺失的保护

**已有的保护机制**：
- `CLAUDE.md` — 架构和约定级别的指引
- `dev-logbook.md` — 过程记录，但偏叙事性
- 测试 — 37 个单元/集成测试，但没有 UI/E2E 测试

**缺失的关键保护**：
- 没有 UI 设计基线（组件布局、交互流程、样式细节）
- 没有 API 契约快照
- 没有"已验证行为"的端到端测试
- 没有产品功能规格文档

### 保护策略选择

**问题**：你倾向哪种保护策略？

| 选项 | 说明 |
|------|------|
| 设计基线文档 | 创建「设计基线」文档，冻结当前已验证的 UI/API/行为设计，未来修改前必须对照 |
| E2E 测试 + 快照 | Playwright 截图 + API 契约测试 + 行为测试，代码级保护 |
| **两者结合** | **文档作为人类参考 + 测试作为自动化守护，双重保护** |
| 其他想法 | 自己的思路 |

**用户选择**：两者结合——文档 + 测试双重保护。

## 第二个问题：从内部实践到平台能力

### 思维跃迁的瞬间

> "同时这些开发过程应该在我们未来的系统自动为我们的开发者进行辅助，让他们可以对这些事情**无感**，你觉得呢，请帮我把这些思考融合到我们之前的规划中，同时我们之前有一个 baseline 去记录我们的核心规划，当前已经在 v1.2 版本了"

这是整个讨论的**关键转折点**。

用户把讨论从"保护我们自己的设计"提升到了一个更高的层面：**这不应该只是我们的内部实践，而应该成为 Forge 平台提供给所有开发者的能力**。而且这个能力的交付方式是"**无感**"（seamless）——开发者不需要知道有这些保护机制在运行，他们只需要感受到"一切都很顺利"。

### 核心洞察的诞生

> "我们自己踩的坑，就是 Forge 平台应该帮用户自动规避的坑。"

这句话凝结了两个重要的认知：

1. **对内**（短期）：为当前 Web IDE 创建设计基线文档 + E2E 测试守护
2. **对外**（平台能力）：将"设计基线冻结 + 自动回归检测"编码为 Forge 的核心 Skill 和底线能力

检视已有的 baseline v1.2：
- 底线体系（4.4 节）已有 5 个底线脚本，但**全部面向代码质量**，没有面向设计保真度的保护机制
- 进化环（4.6 节）关注知识增长，但**没有设计守护维度**

这些缺失正好印证了用户的担忧——连规划文档本身都没有覆盖设计保护。

### 用户的指示

> "我们的 logbook 里面有我们这次 phase 1.5 的全面开发过程，你看下，还有哪些可以提升的部分，同时应该融入到我们 forge 平台的"

## 四大平台能力分析

从 logbook 的真实数据中提炼出 4 大类平台能力：

### 能力 1：Pre-flight Check（部署前飞行检查）

**来源问题**：10 个额外问题中有 7 个可以在构建前检测到

| logbook 问题 | Pre-flight 应检测到 |
|---|---|
| 缺 `package-lock.json` | lockfile 缺失 |
| `ws://` 在 Next.js rewrite 中非法 | 框架配置校验 |
| `.dockerignore` 与 Dockerfile 矛盾 | 部署配置一致性 |
| 无 `public` 目录但 Dockerfile COPY | 构建产物存在性 |
| TypeScript build 错误 | 生产构建预验证 |
| JDK 版本不匹配 | 工具链版本检查 |
| `settings.gradle.kts` 缺 pluginManagement | 构建配置完整性 |

**平台设计**：
- **新 Skill**：`deployment-readiness-check` — 在任何部署操作前自动触发
- **新底线**：`deployment-preflight-baseline.sh` — 阻断不满足前置条件的部署
- **开发者无感**：SuperAgent 在执行部署相关任务时，OODA 的 Orient 阶段自动加载此 Skill，开发者只看到"已检查，可以部署"或"发现 3 个问题需修复"

### 能力 2：Build-Run Gap Detection（编译-运行鸿沟检测）

**来源问题**：编译通过 + 测试通过 ≠ 能启动

| logbook 问题 | 检测维度 |
|---|---|
| `WebClient` bean 缺失 → 启动崩溃 | Spring bean 依赖完整性 |
| 请求体 `{ message }` vs DTO 期望 `{ type, content }` | 前后端契约一致性 |
| Actuator 404 | 端点可达性验证 |
| `npm run dev` 不报的错 `npm run build` 才报 | dev/build 行为差异 |

**平台设计**：
- **增强现有 Skill** `api-design` — 加入"前后端请求/响应体自动对账"能力
- **新底线** `runtime-health-baseline.sh` — 启动应用 → 验证关键端点可达 → 检查 bean 注入完整性
- **进化环集成**：每次 CI/CD 执行的 build-run gap 数据进入 execution-logger，趋势分析哪些模块容易出现"编译通过但运行失败"

### 能力 3：Environment Parity Validation（环境一致性验证）

**来源问题**：本地 vs Docker vs 生产的环境差异

| logbook 问题 | 环境差异类型 |
|---|---|
| Docker 内 TLS 握手失败 | 网络层差异 |
| 本地 JDK 8 vs Docker JDK 21 | 工具链版本差异 |
| `docker-compose.yml` `version` 字段过时 | 配置格式差异 |
| Spring Security 在测试中禁用但运行时拦截 | 安全配置差异 |

**平台设计**：
- **增强 `codebase-profiler`** — 增加"环境画像"维度：扫描项目中的工具链声明，与本地实际环境对比
- **新 Skill** `environment-parity` — 在跨环境部署前检测版本偏差
- **HITL 点**：当检测到关键版本不匹配时（如 JDK 8 vs 21），阻断并提示开发者

### 能力 4：Iterative Debugging Intelligence（迭代调试智能）

**来源问题**：13 次尝试中的策略转换和问题链

| logbook 模式 | 智能化方向 |
|---|---|
| TLS 失败 → 重试 → 再失败 → 策略转换 | **失败模式识别 + 策略推荐**：两次相同错误后自动建议替代方案 |
| `.dockerignore` 排除 → 修复 → COPY 又失败 → 修复 → health check 又失败 | **依赖链分析**：修复一个问题时预测可能引发的下游问题 |
| 每次构建等 6-7 分钟才知道结果 | **快速反馈**：在构建前静态分析 Dockerfile + compose 配置，秒级反馈 |

**平台设计**：
- **增强进化环 `execution-logger`** — 记录"失败 → 修复 → 再失败"的链式模式，沉淀为 troubleshooting 知识
- **增强 `error-handling` Skill** — 加入"部署/构建失败的常见模式库"
- **新能力 `fix-chain-predictor`** — 当修复一个问题时，分析修改的副作用，预测可能引发的下一个问题

## 关键推演：从 90 分钟到 5 分钟

如果 Forge 平台具备以上 4 个能力，Phase 1.5 的调试过程可能是：

```
开发者: "帮我部署到 Docker"

SuperAgent (Orient/Pre-flight):
  ✗ 检测到 package-lock.json 缺失 → 自动生成
  ✗ 检测到 next.config.ts ws:// rewrite 不合法 → 自动修复
  ✗ 检测到本地 JDK 8，项目需要 JDK 21 → 提示切换
  ✗ 检测到 KnowledgeController 注入 WebClient 但无 @Bean → 自动添加
  ✗ 检测到 Dockerfile COPY public/ 但目录不存在 → 自动调整
  ✓ 所有 pre-flight 通过

SuperAgent (Act):
  → 本地构建 (5s backend + 10s frontend)
  → Docker 打包 (15s)
  → 启动验证 (30s)
  ✓ 3 容器 healthy, http://localhost:9000 返回 200

总耗时: ~5 分钟 (vs 实际 90 分钟)
```

这个推演不只是一个数字游戏。它揭示了一个核心真理：**大部分调试时间不是花在解决问题上，而是花在发现问题上**。Pre-flight check 的价值在于把"发现"从分钟级（等构建完成才知道）压缩到秒级（构建前就知道）。

## 回顾：讨论的演化弧线

```
"设计会被遗忘"
    │  用户的情感触发：试用后的喜悦 + 对未来退化的焦虑
    ▼
"文档 + 测试双重保护"
    │  解决方案：冻结当前设计 + 自动化守护
    ▼
"让开发者无感"                    ← 关键跃迁
    │  从"保护我们的设计"到"这应该是平台能力"
    ▼
"从 Logbook 挖掘痛点"
    │  回到真实数据，4 大类问题 → 4 大类平台能力
    ▼
"90 分钟 → 5 分钟"
    │  具体推演证明价值
    ▼
"融入 Baseline v1.3"
    └─ 2 个原则 + 3 个底线 + 3 个 Skill + 进化环增强
```

这次讨论本身就是 Forge 双环架构的一次实践：
- **交付环**产出了 Docker 部署（90 分钟的调试）
- **进化环**把这 90 分钟的痛苦转化为 4 大类平台能力设计

> **交付环中的每一次挫折，都是进化环的养料。**
