# Forge 交付过程分析：方法论、优缺点与 Claude Code 最佳实践

> 基于 18 个 Session 开发日志（2026-02-16 ~ 2026-02-20）的全量分析。
> 数据来源：`docs/planning/dev-logbook.md`（2720 行）、`docs/metrics-report-phase1.6.md`、`docs/phase1.6-e2e-acceptance-test.md`

---

## 一、交付方法论总结

纵观 18 个 Session（2026-02-16 ~ 2026-02-20，约 5 天），交付方法论可概括为 **"文档驱动 + Phase 迭代 + 三纪律守护"**。

### 1.1 Phase 迭代模型

```
Phase 0 (骨架) → Phase 1 (真实功能) → Phase 1.5 (Docker 部署)
→ Phase 2 (OODA+MCP+评估) → Phase 1.6 (AI 交付闭环+SSO)
```

每个 Phase 有明确的验收标准，Phase 结束时做验收测试 + 基线更新。这种模式类似 **时间盒迭代（Time-boxed Iteration）**，但每个 Phase 的大小不固定，由功能聚合度决定。

### 1.2 Session 作为最小工作单元

每个 Session = 一次连贯的 Claude Code 对话，约 2-3 小时。Session 结构高度一致：

```
目标声明 → 实施（代码/文档） → Bug 修复 → 文件变更表 → 经验沉淀 → 统计快照 → Git 提交
```

这是一个 **微型 PDCA 循环**：Plan（目标）→ Do（实施）→ Check（测试/验收）→ Act（经验沉淀/基线更新）。

### 1.3 双基线守护系统

```
规划基线（baseline-v1.x.md）：我们要造什么 → 设计驱动
设计基线（design-baseline-v1.md）：我们造了什么 → 实现驱动
```

两份文档独立演进，Phase 结束后交叉校验。这是一种 **规格-实现一致性保障机制**，在传统软件工程中相当于需求追溯矩阵（Requirements Traceability Matrix），但更轻量。

### 1.4 验收测试驱动

从 Phase 2 开始引入结构化验收测试文档（最终 21 场景 / 87 用例），既是产品规格也是可执行校验。Session 14-18 的五个 Session 专门用于系统执行验收测试 + 修 Bug + 校准文档，这是典型的 **验收测试驱动开发（ATDD）** 模式。

## 二、优点

### 2.1 文档纪律极强，有效对抗 AI 遗忘

18 个 Session 的 logbook 是项目最有价值的资产。每个 Session 的文件变更表、Bug 记录、经验沉淀都非常完整。这直接解决了 Claude Code 最大的痛点：**上下文丢失**。新 Session 启动时，通过 logbook + 基线文档 + CLAUDE.md 三件套，Claude 可以快速恢复上下文。

**数据佐证**：18 个 Session 横跨至少 3-4 次 context 重置，但项目从未因此中断或倒退。

### 2.2 Bug 追踪闭环

20 个 Bug 全部有编号、等级、根因、修复方案。关键的是 **根因分析** 做得好：

- BUG-008 和 BUG-017 识别为同类问题（Kotlin 枚举序列化）
- BUG-012 揭示了 REST API 和 WebSocket 两条路径的独立性
- BUG-014 和 BUG-020 识别为焦点管理的矛盾对

这种模式让系统性缺陷可追踪、可预防。

### 2.3 统计快照提供客观进度度量

每个 Session 末尾的统计快照（文件数、测试数、Bug 数、通过率等）让进度可量化：

| 指标 | Session 1 | Session 11 | Session 18 | 增长 |
|------|-----------|------------|------------|------|
| 文件数 | 227 | 280+ | 325+ | +43% |
| 代码行数 | 37K | 47K | 50K+ | +35% |
| 单元测试 | 0 | 128+ | 147 | 从零到有 |
| Git Commits | 1 | 18 | 33+ | — |
| MCP 工具 | 0 | 6 | 9 | — |
| Bug 发现/修复 | 0/0 | 0/0 | 20/19 | — |
| E2E 通过率 | — | 22/24 | 80/87 (92%) | — |

### 2.4 经验沉淀可编码

logbook 中的"经验沉淀"章节直接指导了 CLAUDE.md 的进化（67 行 → 181 行）和开发纪律的形成：

| 来源 Session | 经验 | 编码位置 |
|-------------|------|---------|
| Session 14 | 网络依赖是自动化的敌人 | CLAUDE.md "本地构建 + Docker 打包" |
| Session 15 | 枚举序列化是隐蔽杀手 | CLAUDE.md 已知陷阱 |
| Session 13 | 验收测试编写后必须对照代码做交叉验证 | 开发纪律第三支柱 |
| Session 14 | REST API 和 WebSocket 是两条独立路径 | CLAUDE.md 已知陷阱 |
| Session 15 | 多轮 Docker 重建的时间成本 | CLAUDE.md Docker 部署注意事项 |

### 2.5 并行 Agent 利用

Session 1 同时启动 6 个 Agent 创建不同模块，虽然遇到 rate limit 和权限问题，但一次性创建了 227 文件 / 37,179 行。这是 Claude Code 的核心优势之一。

## 三、可改进的点

### 3.1 Phase 编号反复变更，暴露规划粒度问题

Phase 2 完成后出现了 "Phase 2.5"，后来又改为 "Phase 1.6"（Session 13 做了 69 处全局重命名）。这说明初始 Phase 规划的粒度可能太粗，中间阶段的功能归属不明确。

**建议**：改用 Milestone 或语义化版本号（v0.1 骨架 → v0.2 真实功能 → v0.3 部署 → v0.4 OODA → v0.5 交付闭环），避免 Phase 编号的混乱。

### 3.2 验收测试"先写后验"的滞后

验收测试在 Session 12 编写（89 用例），但到 Session 13 交叉验证时发现了 **16 个错误**。这意味着如果不做交叉验证，这些测试本身就是错误的。

**根因**：验收测试写在代码完成后，而非代码完成前。真正的 ATDD 应该是先写验收用例 → 再写代码 → 最后执行。当前流程更接近"验收测试后补"。

**建议**：下一个 Phase 尝试在编码前写验收场景（至少写场景标题和关键预期），编码后再填充细节。

### 3.3 Docker 重建循环效率低

Session 15 提到"每次改一行代码都需要 build → deploy → wait → test → 发现新问题 → 重复"，4 次 Docker 重建消耗了大量时间。Session 14 的 JDK 安装尝试了 4 种方式全部失败。

**这是一个典型的反馈循环太长的问题**。

| 反馈循环 | 耗时 | 频率 |
|---------|------|------|
| 单元测试 | ~30s | 每次修改 |
| `npm run build` 类型检查 | ~10s | 每次修改 |
| Docker 重建 + 部署 | ~3-5min | 每次集成验证 |
| 89 用例全量验收 | ~2h | 每个 Phase |

**建议**：建立本地开发模式（`./gradlew bootRun` + `npm run dev`），把大部分 Bug 发现环节提前到 Docker 之前。Docker 只用于最终集成验证。

### 3.4 同类 Bug 反复出现

枚举序列化问题出现了两次（BUG-008 FileType/WorkspaceStatus、BUG-017 ServiceType/ServiceStatus）。Session 16 logbook 已经意识到"应全局排查所有枚举"，但排查并未在当时执行。

**建议**：发现系统性 Bug 时，立即做全局排查并一次性修复（`grep -r "enum class" --include="*.kt"`），而不是等下一次踩坑。

### 3.5 文档量大但维护成本高

18 个 Session 产出的文档总量：

| 文档 | 行数 | 更新频率 |
|------|------|---------|
| dev-logbook.md | 2720 | 每个 Session |
| phase1.6-e2e-acceptance-test.md | 1059 | Session 12-18 |
| design-baseline-v1.md | 1007 | 每个 Phase |
| baseline-v1.4.md | 815 | 每个 Phase |
| metrics-report-phase1.6.md | ~300 | Phase 结束 |
| buglist.md | ~200 | 持续 |
| **总计** | **~6100** | — |

Session 13 专门花了一整个 Session 做文档重构，Session 18 又做了验收测试重构。**文档债务的积累速度超过了预期**。这是文档驱动方法论的固有代价——如果不做定期重构，文档反而会成为负担而非资产。

**建议**：每 5 个 Session 安排一次文档精简（类似 Session 13），或考虑用自动化脚本从代码生成部分文档（如 MCP 工具清单、API 端点列表）。

## 四、与 Claude Code 最佳实践的对比

### 4.1 CLAUDE.md 使用 — ★★★★★ 优秀

CLAUDE.md 从 67 行进化到 181 行，编码了：
- 交互偏好（中文、行动优先）
- 架构知识（模块依赖、容器部署）
- 已知陷阱（7 条，从 14 个 Session 提炼）
- 开发纪律（三大支柱）

这是 Claude Code 官方推荐的核心实践：**把项目知识编码到 CLAUDE.md 中**，让每个新 Session 自动继承。

**超越之处**：大多数 Claude Code 用户的 CLAUDE.md 只有 10-20 行基本配置。181 行的 CLAUDE.md 编码了完整的项目知识体系，这在 Claude Code 用户中属于顶级实践。

### 4.2 Plan Mode 使用 — ★★★★ 良好

Session 1 和 Phase 2 规划都用了 Plan Mode。但中间的 Sessions（6-12）大多是直接执行，没有经过 Plan Mode。对于 Phase 1.6 这样的大型功能集（8 大功能块 / 30 个文件 / 3190 行），理想情况下应该先 Plan 再执行。

**Claude Code 推荐**：任何涉及 5+ 文件修改的任务都值得先 Plan。

### 4.3 并行 Agent（Task 工具）— ★★★☆ 有潜力未充分释放

Session 1 充分利用了并行 Agent（6 个同时跑），但后续 Sessions 基本回到了单线程模式。

**可以用并行 Agent 的场景（未使用）**：
- Session 12：前后端 30 个文件可以分两个 Agent 同时写
- Session 14-17：不同场景的验收测试可以并行执行
- 文档更新：logbook + baseline + acceptance test 可以并行修改

### 4.4 反馈循环速度 — ★★★☆ 有改进空间

Claude Code 最佳实践强调 **快速反馈**：写代码 → 跑测试 → 看结果 → 修复。

| 环节 | 当前 | 最佳实践 |
|------|------|---------|
| 单元测试 | ✅ ~30s | ✅ 快 |
| 前端类型检查 | ✅ `npm run build` | ✅ 好（CLAUDE.md 已编码此经验） |
| 集成测试 | ❌ Docker 重建 3-5min | 本地 `bootRun` + `dev` 更快 |
| E2E 测试 | ❌ 手动 curl + UI | 可 Playwright 自动化 |

### 4.5 Git 提交粒度 — ★★★★ 良好

大多数 commit 有清晰的语义（`feat:` / `fix:` / `docs:`），但有些 Session 的提交太大：

| Commit | 文件数 | 行数 | 评价 |
|--------|--------|------|------|
| `02e003c` Phase 0 | 227 | 37,179 | 初始化，可接受 |
| `4759ee0` Phase 1.6 | 29 | 1,883 | 偏大，可拆为 3-4 个 |
| `9b6f62e` 11 Bug | ~8 | ~500 | 偏大，可按 Bug 拆 |

**Claude Code 推荐**：每个逻辑变更一个 commit，便于 bisect 和回滚。

### 4.6 持久知识管理 — ★★★★★ 开创性

"三纪律"系统（Logbook + 双基线 + 验收测试）本质上是在 Claude Code 之上构建了一套 **项目记忆系统**。这超越了 CLAUDE.md 的范畴。

在 Claude Code 的标准实践中，CLAUDE.md 是唯一的跨 Session 记忆载体。三纪律系统补充了：

| 记忆维度 | 载体 | 回答的问题 |
|---------|------|-----------|
| 时间维度 | Logbook | 过去发生了什么？为什么这样做？ |
| 空间维度 | 设计基线 + 规划基线 | 系统当前的全貌是什么？ |
| 质量维度 | 验收测试 + Buglist | 什么能工作？什么不能？ |
| 知识维度 | CLAUDE.md + 经验沉淀 | 我们学到了什么？ |

这套系统对于需要跨越多个 Session（>10 次对话）的长期项目具有显著价值，值得输出为可复用的方法论模板。

## 六、方法论一句话总结

> **"用文档对抗 AI 遗忘，用验收测试对抗质量腐化，用双基线对抗设计偏移。"**

这在 Claude Code 生态中是一套成熟的长期项目实践，核心洞察是：**Claude Code 的上下文窗口是有限的，但项目的复杂度是无限增长的。因此必须建立外部记忆系统来弥补 AI 的遗忘，并用纪律来保证记忆的准确性。**
